## Context

目标是构建一个可复用的组合回测实验系统，使用户可以在不同资产、不同区间、不同约束下快速比较组合表现。当前缺少统一的资产输入标准、组合表达方式、回测执行规则和结果分析口径，导致实验结果不可复现、难横向比较。

本设计面向“研究与验证”场景，不涉及实盘交易执行。核心约束包括：
- 多市场资产交易日历不同（A 股、美股、加密等）。
- 存在跨币种估值需求（CNY、USD 等）。
- 用户需要批量实验并比较结果稳定性，而不仅是单次回测。

## Goals / Non-Goals

**Goals:**
- 提供统一资产池模型，标准化资产元数据与价格序列输入。
- 提供组合定义模型，支持固定权重与参数化组合生成，并支持约束校验。
- 提供确定性的回测执行流程，支持时间区间、调仓频率、成本模型和基准货币。
- 提供统一分析输出，覆盖收益、波动、最大回撤与风险收益指标，支持批量排名与区间对比。

**Non-Goals:**
- 不在回测执行阶段实时调用外部行情接口。
- 不实现券商交易下单。
- 不覆盖复杂衍生品定价（期权希腊值、保证金风控等）。
- 不在 v1 解决高频/分钟级回测，仅支持日级回测口径。

## Decisions

### 1) 采用“声明式实验规格”作为统一输入
- 决策：将用户输入拆为 `UniverseSpec`、`PortfolioSpec`、`BacktestSpec` 三类声明式配置。
- 原因：便于序列化、复现与批处理；同一组合可在多区间重复运行。
- 备选：
  - 仅提供命令行参数：灵活但难复现复杂实验。
  - 直接写代码定义实验：能力强但门槛高，且不利于标准化结果对比。

### 2) 回测时序以“调仓日估值快照”驱动
- 决策：以调仓频率生成目标调仓日集合，在每个估值日按可用收盘价进行组合估值与再平衡。
- 原因：对日级回测最稳健，且可跨市场统一执行。
- 备选：
  - 严格共同交易日交集：会丢失大量信息。
  - 各市场独立估值再合并：实现复杂且可解释性差。

### 3) 组合生成与回测执行解耦
- 决策：先由构建模块产出候选组合集合，再由引擎逐个执行回测。
- 原因：允许固定权重、网格搜索、随机采样共用同一执行引擎；便于后续并行计算。
- 备选：
  - 在引擎内临时生成组合：耦合高，测试困难。

### 4) 统一基准货币估值并保留原币种信息
- 决策：回测结果以单一基准货币输出净值曲线，同时记录资产原币种与汇率使用轨迹。
- 原因：跨币种组合必须在同一口径比较风险收益。
- 备选：
  - 仅原币种并列输出：无法直接聚合组合净值。
  - 固定单币种资产池：限制系统通用性。

### 5) 分析层采用“核心指标 + 可扩展指标”结构
- 决策：内置累计收益、年化收益、年化波动、最大回撤、Sharpe/Sortino/Calmar；允许后续扩展指标插件。
- 原因：先满足常见配置研究需求，同时保留扩展空间。
- 备选：
  - 一次性内置大量指标：初期复杂度过高，维护成本大。

### 6) 采用“在线采集 + 离线快照回测”的数据模式
- 决策：外部 API 仅用于数据采集与更新；回测引擎只读取版本化离线快照（`snapshot_id`）执行。
- 原因：保证结果可复现、可审计，并降低限流、超时与网络抖动对回测稳定性的影响。
- 备选：
  - 回测过程中实时拉取行情：实现简单但结果不稳定，且难以回归验证。
  - 完全手工导入离线文件：可复现但运营成本高，数据更新效率低。

### 7) 采用“准确优先”的默认缺失数据策略
- 决策：对“应有交易日”缺失价格或缺失 FX 默认失败（fail-fast）；仅在资产市场休市时允许沿用最近可用收盘价做估值延续，并标记为 stale valuation。
- 原因：前值填充会掩盖数据缺口并引入偏差，fail-fast 更符合研究准确性要求。
- 备选：
  - 全局前值填充：运行稳定但会引入潜在偏差。
  - 自动插值：可读性更高但统计假设过强，难以审计。

## Risks / Trade-offs

- [跨市场日历错配导致估值偏差] -> 在结果中暴露每次估值所用价格日期，并提供“缺失价格处理策略”字段。
- [跨币种汇率口径不一致] -> 强制要求实验声明汇率来源与时间戳口径，并写入实验元数据。
- [批量组合规模过大导致运行时间不可控] -> 增加组合数上限与分批执行策略，输出进度与中间结果。
- [用户误用参数导致过拟合] -> 提供多区间对比报告并提示“训练区间/验证区间”分离。
- [快照更新节奏不当导致数据陈旧或频繁漂移] -> 制定快照发布节奏并在结果中强制展示 `snapshot_id` 与数据时间戳。
- [严格失败策略导致部分实验无法执行] -> 提供清晰的缺失数据报告与补数流程，不降低默认准确性门槛。

## Migration Plan

1. 先实现在线采集到标准化数据存储与离线快照发布。
2. 基于 `snapshot_id` 实现离线单机回测流程（单区间 + 批量组合）。
3. 补充多区间批处理与统一结果模型。
4. 增加稳健性分析与排名导出能力。
5. 后续再引入更多数据源适配器与并行执行优化。

本变更为新能力引入，不涉及存量线上数据迁移；若后续引入持久化，可通过版本化结果结构兼容旧输出。

## Open Questions

- 交易成本模型在 v1 是否默认按单一费率，还是区分资产类别费率模板？

## Default Policy (Accuracy-First)

- 快照发布：`每日增量采集 + 每周冻结快照`（周冻结作为研究默认基线）。
- 回测输入：必须显式指定 `snapshot_id`，且该快照已通过完整性校验。
- 价格缺失：
  - 应有交易日缺失：默认失败并返回缺失清单。
  - 市场休市：允许估值沿用最近收盘价并标记 stale，不触发该资产调仓成交。
- FX 缺失：
  - 需换汇且应有交易日缺失：默认失败。
  - 不需换汇（资产币种等于基准币种）：无需 FX 序列。
- 偏差控制：默认禁用对“应有交易日缺失”的前值填充与插值。
